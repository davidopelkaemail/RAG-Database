What is RAG?

RAG (Retrieval Augmented Generation) is a technique that combines the power of large language models with external knowledge bases to provide more accurate and contextual responses.

Key Components

  • 📄 Documents: Your knowledge base (PDFs, MD files, text documents)
  • ✂️ Chunking: Splitting documents into smaller, manageable pieces
  • 🎯 Embedding: Converting text into numerical vectors (arrays of numbers)
  • 💾 Vector Database: Specialized storage for vector similarity search
  • 🔍 Retrieval: Finding the most relevant chunks for a query
  • 🤖 Generation: LLM uses retrieved context to generate accurate answers

Why RAG?

  • Up-to-date: Add new documents without retraining the model
  • Accurate: Reduces hallucinations by grounding responses in actual data
  • Private: Keep sensitive data local while using powerful LLMs
  • Efficient: Cheaper than fine-tuning for domain-specific knowledge

Interactive Chunking Demo

Chunking is the process of splitting large documents into smaller pieces (chunks) that can be processed and stored efficiently.

📄 Source Text

RAG (Retrieval Augmented Generation) is a technique that combines large language models with external knowledge bases. The process involves several steps. First, documents are split into smaller chunks. Each chunk is then converted into a vector using an embedding model. These vectors are stored in a specialized database called a vector database. When a user asks a question, the query is also converted to a vector. The system then searches for the most similar vectors in the database. The retrieved chunks provide context for the language model to generate accurate responses.

Chunking Strategies

  • Fixed-size: Split by character count with optional overlap
  • Sentence-based: Split at sentence boundaries
  • Paragraph-based: Keep paragraphs together
  • Semantic: Use embeddings to find natural breaks
  • Recursive: Split by multiple separators hierarchically
⚠️ Important: Smaller chunks = more precise retrieval but may lose context. Larger chunks = more context but may include irrelevant information.

How Embeddings Work

Each word/text is converted into a vector with hundreds of dimensions. The model learns these representations by analyzing billions of texts.

// Example embedding (simplified to 5 dimensions)
"cat" → [0.2, -0.5, 0.8, 0.1, -0.3]
"dog" → [0.3, -0.4, 0.7, 0.2, -0.2]
"car" → [-0.5, 0.8, -0.2, 0.9, 0.4]

// Similar words have similar vectors!
// cat ↔ dog: high similarity (both are animals)
// cat ↔ car: low similarity (unrelated concepts)

Popular Embedding Models

  • nomic-embed-text: 768 dimensions, open source, great for RAG
  • text-embedding-ada-002: OpenAI's model, 1536 dimensions
  • BERT: 768 dimensions, classic transformer model
  • Sentence-BERT: Optimized for sentence similarity

How Storage Works

Key Concepts

  • Vector Index: Data structure for fast similarity search (HNSW, IVF, LSH)
  • Distance Metrics: Cosine similarity, Euclidean distance, dot product
  • Metadata Filtering: Filter by source, date, author before vector search
  • Hybrid Search: Combine keyword search with vector similarity

LanceDB Example (OpenClaw)

// OpenClaw uses LanceDB for vector storage
// Path: ~/.openclaw/memory/lancedb
// Table: memories_indexed
// Embedding: nomic-embed-text (768 dims)

// Store vectors
await table.add([
  { text: "chunk content", embedding: [0.1, 0.2, ...] }
]);

// Search similar
const results = await table
  .vectorSearch(queryEmbedding)
  .limit(5)
  .toArray();

Retrieval Strategies

  • Top-K: Return K most similar chunks
  • Threshold: Only return chunks above similarity threshold
  • MMR: Maximal Marginal Relevance - diversity in results
  • Hybrid: Combine vector search with keyword search
  • Re-ranking: Use another model to re-order results
